# mymuduo - 高性能多线程 C++ 网络库

基于 Reactor 模式和非阻塞 I/O 实现的高性能 C++ 网络库，旨在提供高并发、低延迟的服务器开发框架。本项目不仅是对经典 Muduo 库的重构与核心功能复刻，更在多核环境下进行了极致的性能压测与优化。

---

## 🚀 核心特性

- **Reactor 模型**：基于 `epoll` 的水平触发（LT）模式，采用多线程 `EventLoop` 异步调度。
- **线程模型**：支持 **One Loop Per Thread**。由主 Loop 负责 Accept 新连接，并通过轮询（Round Robin）分发给子 Loop 线程处理 I/O，充分利用多核性能。
- **异步日志系统**：实现双缓冲异步日志，支持编译期宏剥离（`QPSTESTON`），消除高频压测时的 I/O 阻塞。
- **时间轮心跳检测**：利用 `std::weak_ptr` 与时间轮算法实现 60s 空闲连接自动剔除，解决了传统计时器在多线程下的循环引用与死锁问题。
- **高效缓冲区**：设计了可自动增长的 `Buffer` 类，支持预留空间与高效的数据读取/发送。

## 🧠 关键技术挑战与解决记录

在项目开发与压测过程中，针对暴露出的性能瓶颈与逻辑缺陷，进行了以下核心优化：

### 1. 修复 SO_ERROR: 104 (ECONNRESET) 异常
* **现象描述**：在初期压测阶段，服务器频繁报出 `TcpConnection::handleError() - SO_ERROR:104` 错误，导致 `wrk` 产生大量 Read Errors。
* **原因分析**：原逻辑在 `onWriteComplete` 回调中主动调用了 `conn->shutdown()`。在 `wrk` 使用 Keep-Alive 长连接发包时，服务器刚发完响应就强行关闭写端，导致 TCP 连接被异常重置（RST）。
* **解决方案**：移除服务器端的响应后主动断开逻辑，将连接的生命周期交由客户端或心跳检测机制管理。
* **结果**：彻底消除了 `104` 报错，QPS 从 3.6w 稳定攀升至 **12.8w+**。

### 2. 突破 I/O 性能瓶颈 (编译期脱敏)
* **现象描述**：初始 QPS 仅为 8.8k 左右，且伴随严重的 Latency 抖动。
* **原因分析**：在高频请求下，每一条 `LOG_INFO` 都会触发字符串格式化与磁盘写入操作。即使是异步日志，在每秒十万级的调用下也会产生巨大的系统调用开销和缓冲区争用。
* **解决方案**：引入 `QPSTESTON` 宏。在 `Logger.h` 中通过预处理器判定，若开启该宏，则 `LOG_INFO` 被替换为空操作，实现编译期的“零开销”日志。
* **结果**：排除了 I/O 阻塞干扰，系统算力全额分配给网络事件处理，QPS 实现了 **14 倍** 的飞跃。

### 3. 时间轮死锁与 Poll 精度优化
* **循环引用修复**：
    * **问题**：最初 `TcpConnection` 的 `Context` 存储了 `EntryPtr`（强引用），而 `Entry` 内部又持有 `TcpConnection` 的强引用，导致引用计数永远无法归零，连接无法正常析构。
    * **解决**：将 Context 存储改为 `std::weak_ptr<Entry>`，打破循环引用，确保连接断开时资源能正常释放。
* **阻塞精度调整**：
    * **问题**：`epoll_wait` 原设定阻塞时间为 10s（`kPollTimeMs = 10000`），在低并发或空闲时会导致定时器回调产生巨大的触发延迟。
    * **解决**：将阻塞精度优化至 1s，在保证不频繁空转 CPU 的前提下，确保了心跳检测（60s 超时）的响应实时性。

## 📊 性能表现 (QPS Benchmark)

在 **14 核物理机** 环境下，使用 `wrk` 进行 HTTP 长连接压测，本项目展现出了极高的吞吐量与纳秒级的响应稳定性。

- **操作系统**: Ubuntu 22.04 LTS
- **硬件配置**: 14-Core Intel Processor / 32GB RAM
- **网络环境**: 本地回环 (127.0.0.1)
- **编译器**: g++ 11.4.0 (优化选项: `-O3`)
